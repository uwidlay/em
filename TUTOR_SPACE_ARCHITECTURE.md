## 1. Архитектурная позиция для MVP

Рекомендация: строить MVP как React-приложение с Supabase в роли backend/data-платформы и Vercel в роли хостинга фронтенда.

Почему это подходит домену:

- в ТЗ уже выбран стек React + Supabase + Vercel;
- у продукта нет сложной серверной бизнес-логики вне CRUD, доступа, загрузки фото и статусов Д/З;
- Supabase закрывает Auth, Postgres, Storage и RLS;
- Vercel хорошо подходит для быстрого деплоя frontend-приложения;
- MVP должен быстро показать связку "репетитор - ученик - урок - Д/З - фото - проверка".

Ключевой принцип: не строить отдельный backend раньше времени. Сначала использовать Supabase как backend-as-a-service, а серверные функции добавлять только там, где клиенту нельзя доверять секреты или нужно безопасно выполнить действие.

- отдельную CMS;

- платежную систему;

- полноценную CMP-платформу, если нет рекламных/аналитических cookies.

## 5. Backend/data-архитектура на Supabase

Supabase используется как:

- Auth для репетиторов;
- Postgres для доменных данных;
- Storage для фото Д/З;
- RLS для изоляции данных;
- возможно RPC / Edge Function для операций с токеном ученика.

### 5.1. Таблицы

Базовая схема соответствует ТЗ:

- `tutors`;
- `students`;
- `useful_links`;
- `lessons`;
- `lesson_materials`;
- `homework_submissions`;
- `homework_submission_photos`;
- `update_events` или упрощенное поле `students.has_unread_updates_for_student`.

Рекомендация для MVP: начать с таблицы `update_events`, потому что UI теперь показывает короткий список 3-5 последних обновлений ученика. Это не полноценный центр уведомлений, но требует хранить несколько последних событий и отмечать их просмотренными.

Раздел "Расписание" в текущем MVP остается визуальным календарем на стороне frontend/product prototype. Он не является источником уроков, Д/З или групповых занятий и не требует отдельной таблицы в БД MVP без нового согласования доменной модели.

### 5.3. Доступ к данным

Репетитор работает как authenticated user Supabase Auth.

Ученик работает без Supabase Auth-сессии. Для ученика нужен отдельный безопасный механизм доступа по токену:

- токен хранится у ученика в URL;
- в базе хранится хэш токена, не открытый токен;
- при открытии `/student/:token` приложение получает данные профиля через безопасный слой, который проверяет токен;
- перегенерация ссылки меняет токен и инвалидирует старую ссылку.

## 6. Авторизация репетитора и доступ ученика по токену

### 6.1. Репетитор

Использовать Supabase Auth:

- sign up с email, password и metadata для имени/телефона или последующим созданием `tutors`;
- email confirmation включен;
- sign in по email/password;
- password recovery через email.

После регистрации в `tutors` должен быть профиль, связанный с `auth.users.id`.

Рекомендация: создать `tutors.auth_user_id` и использовать его в RLS-политиках.

### 6.2. Ученик

Ученик не должен становиться Supabase Auth user в MVP.

Рекомендованный вариант:

1. При создании ученика серверная операция генерирует случайный токен.
2. В `students.access_token_hash` сохраняется хэш токена.
3. Репетитор получает исходный токен только как URL для копирования.
4. При открытии `/student/:token` приложение вызывает безопасную RPC/Edge Function.
5. RPC/Edge Function хэширует переданный токен, ищет ученика, возвращает только разрешенный набор данных.

Почему не стоит читать `students` напрямую из браузера по anon key:

- RLS с plain token в клиентском запросе легко сделать ошибочно;
- нельзя хранить открытые токены;
- нужно централизованно контролировать, какие поля видит ученик;
- фото требуют аккуратной выдачи доступа.

Минимальный безопасный подход для MVP: Supabase Edge Function или Postgres RPC `security definer` для student public profile. Конкретный вариант нужно подтвердить при реализации.

## 7. RLS-подход

Официальная позиция Supabase: RLS нужно включать на таблицах в exposed schema, а политики работают как добавляемое к запросам условие доступа. Storage также управляется через RLS на `storage.objects`.

### 7.1. Общие правила

- Включить RLS на всех доменных таблицах.
- Для authenticated tutor разрешать доступ только к данным, где `tutor_id` соответствует текущему `auth.uid()` через связь `tutors.auth_user_id`.
- Для ученика не открывать прямые RLS-политики на все таблицы без необходимости; использовать безопасный слой по токену.
- Service role key никогда не отдавать в браузер.
- Для `INSERT`/`UPDATE` использовать не только `using`, но и `with check`, чтобы пользователь не мог записать данные в чужую область.

### 7.2. Политики по таблицам

| Таблица | Репетитор: SELECT | Репетитор: INSERT | Репетитор: UPDATE | Репетитор: DELETE | Ученик по токену |
|---|---|---|---|---|---|
| `tutors` | Только свой профиль. | Только профиль, связанный с auth user. | Только свой профиль. | Не нужно в MVP. | Нет прямого доступа. |
| `students` | Только свои ученики. | Только с `tutor_id` своего профиля. | Только свои ученики. | Физический delete не нужен; архивирование через update. | Через RPC/Edge Function возвращать ограниченный профиль. |
| `useful_links` | Только ссылки своих учеников. | Только для своих учеников. | Только для своих учеников. | Только для своих учеников. | Через RPC/Edge Function только ссылки своего профиля. |
| `lessons` | Только уроки своих учеников. | Только для своих учеников. | Только уроки своих учеников. | Только уроки своих учеников или soft delete. | Через RPC/Edge Function только уроки своего профиля. |
| `lesson_materials` | Только материалы уроков своих учеников. | Только для уроков своих учеников. | Только для уроков своих учеников. | Только для уроков своих учеников. | Через RPC/Edge Function только материалы своего профиля. |
| `homework_submissions` | Только сдачи своих учеников. | Обычно ученик через токен; репетитору insert не нужен. | Репетитору прямой update не нужен, проверка хранится в `lessons`. | Не нужно в MVP. | Создание сдачи через RPC/Edge Function. |
| `homework_submission_photos` | Только фото сдач своих учеников. | Через загрузочный поток сдачи Д/З. | Не нужно в MVP. | Не нужно в MVP. | Через RPC/Edge Function / signed URL только свои фото. |
| `update_events` | Только события своих учеников. | Системные операции при изменениях. | Отметить просмотр учеником через токен. | Не нужно в MVP. | Через RPC/Edge Function только свои события/mark seen. |

### 7.3. Удаление урока

Принятое решение: использовать soft delete через `deleted_at` для уроков.

Причины:

- в схеме поле уже предусмотрено;
- случайное удаление урока не разрушит историю мгновенно;
- это соответствует осторожному подходу к учебной истории.

RLS-политики и запросы списка уроков должны по умолчанию исключать строки, где `deleted_at is not null`.

## 8. Хранение фото Д/З в Supabase Storage

### 8.1. Bucket

Рекомендация:

```text
Bucket: homework-photos
Public: false
```

Фото должны храниться приватно.

### 8.2. Структура путей

Рекомендованный путь:

```text
homework-photos/
  tutor_{tutorId}/
    student_{studentId}/
      lesson_{lessonId}/
        submission_{submissionId}/
          {photoId}.{ext}
```

Зачем:

- удобно писать политики и отладочные запросы;
- легко найти файлы конкретной сдачи;
- путь отражает доменную иерархию.

### 8.3. Метаданные в БД

В `homework_submission_photos` хранить:

- `submission_id`;
- `storage_path`;
- `original_filename`;
- `mime_type`;
- `size_bytes`;
- `width`;
- `height`;
- `sort_order`;
- `created_at`.

### 8.4. Загрузка

Поток:

1. Ученик выбирает до 10 фото.
2. Клиент проверяет количество, формат JPG/PNG/HEIC/HEIF/WebP и лимит исходного фото до 10 МБ.
3. Клиент сжимает фото до 1600-2000 px по длинной стороне; целевой размер после сжатия желательно 1-2 МБ.
4. Создается запись сдачи Д/З.
5. Фото загружаются в приватный bucket.
6. Создаются записи `homework_submission_photos`.
7. Статус урока становится `in_review`.
8. Для репетитора появляется бейдж "Д/З на проверке".

Архитектурное решение, которое нужно подтвердить: делать шаги 4-8 через Edge Function/RPC, чтобы атомарно связать сдачу, фото и статус.

### 8.5. Доступ к просмотру фото

Репетитор:

- видит фото только своих учеников.

Ученик:

- видит фото своих сдач, если интерфейс предусматривает просмотр отправленного.

Рекомендация:

- не делать bucket публичным;
- выдавать signed URLs на ограниченное время или проксировать доступ через безопасную функцию;
- не хранить публичные URL в базе.

## 10. CMP / cookie consent

CMP здесь понимается как Consent Management Platform / механизм согласия на cookies и обработку данных.

### 10.1. Что есть в MVP

В MVP есть:

- Supabase Auth session;
- технические cookies/local storage/session storage, необходимые для входа репетитора;
- возможно, хранение сессии Supabase в браузере;
- персональные данные и фото Д/З.

В MVP не запланированы:

- рекламные cookies;
- маркетинговая аналитика;
- поведенческий трекинг;
- сторонние рекламные пиксели.

### 10.2. Нужна ли полноценная CMP

Рекомендация: полноценную внешнюю CMP-платформу отложить для MVP, если не используются рекламные/аналитические cookies.

Минимально для MVP:

- подготовить политику конфиденциальности;
- описать использование технических cookies/локального хранения для авторизации;
- при регистрации дать ссылку на документы и согласие с условиями;
- если добавляется аналитика, вернуться к вопросу cookie banner / consent banner.

Важно: юридическую финальную формулировку должен проверить юрист, особенно если сервис запускается публично и обрабатывает данные учеников/родителей.

### 10.3. Что не делать в MVP

- не внедрять тяжелую CMP ради несуществующих маркетинговых cookies;
- не добавлять аналитику "просто на всякий случай";
- не обещать пользователю управление категориями cookies, если есть только технические cookies.

## 11. Деплой на Vercel

### 11.1. Рекомендация по проекту

Вариант:

- frontend-приложение деплоится на Vercel;
- Supabase остается отдельным managed backend;
- переменные окружения задаются в Vercel per environment.

Официальная документация Vercel указывает, что environment variables задаются вне исходного кода, могут различаться по environments и применяются к новым deployment после изменения.

### 11.2. Environment variables

Минимально:

```text
VITE_SUPABASE_URL
VITE_SUPABASE_ANON_KEY
```

Если используются server-side функции:

```text
SUPABASE_SERVICE_ROLE_KEY
SUPABASE_JWT_SECRET
```

Правила:

- `VITE_SUPABASE_ANON_KEY` может быть в клиенте, но все равно требует RLS;
- `SUPABASE_SERVICE_ROLE_KEY` никогда не должен попадать в браузер;
- production/preview/development переменные должны быть разделены;
- изменения переменных требуют нового deployment.

### 11.3. Preview deployments

Рекомендация:

- использовать preview deployments для проверки экранов и сценариев;
- держать отдельный Supabase project для production;
- для preview/dev можно использовать отдельный Supabase project или отдельные данные, чтобы не смешивать реальные данные и тесты.

### 11.4. Production checklist MVP

Перед production:

- RLS включен на всех доменных таблицах;
- Storage bucket приватный;
- service role key не доступен клиенту;
- email confirmation включен;
- password recovery работает;
- правила доступа ученика по токену проверены;
- загрузка фото проверена на mobile;
- юридические документы готовы или явно отмечены как блокер публичного запуска.

## 14. Архитектурные риски

| ID | Риск | Почему важен | Рекомендация |
|---|---|---|---|
| AR-001 | Неверная RLS-политика откроет чужие данные. | SaaS требует строгой изоляции репетиторов. | Писать RLS с тестами доступа и принципом deny by default. |
| AR-002 | Ученический токен будет реализован слишком открыто. | Ученик входит без пароля, токен фактически является ключом доступа. | Хранить только хэш токена, проверять через безопасный слой, инвалидировать старый токен. |
| AR-003 | Service role key попадет в клиент. | Это обходит RLS и ломает безопасность. | Хранить service key только в server-side окружении, не использовать с `VITE_`. |
| AR-004 | Фото будут публичными. | Фото Д/З могут содержать персональные данные. | Только private bucket, RLS/signed URLs. |
| AR-005 | Загрузка фото будет ненадежной на смартфоне. | Сдача Д/З с фото - ключевой сценарий MVP. | Клиентское сжатие, понятный прогресс, повторная попытка, тесты на mobile. |
| AR-006 | Модальное окно урока станет перегруженным. | У урока много полей. | Группировать поля и не добавлять вне-MVP функции. |
| AR-007 | Логика актуального Д/З будет спорной. | В ТЗ есть открытый вопрос для нескольких несданных Д/З. | Подтвердить алгоритм до реализации. |
| AR-008 | История повторных сдач будет непонятна. | В базе история есть, интерфейс открыт. | Подтвердить: показывать только последнюю или все сдачи. |
| AR-009 | Юридические документы отложат запуск. | Есть персональные данные и фото. | Отметить как production blocker для публичного запуска. |
