# TUTOR-SPACE: техническое задание MVP

## 1. Общее описание

TUTOR-SPACE - минималистичная SaaS-платформа для совместной работы репетитора и ученика. Сервис заменяет Google Таблицы в сценариях ведения уроков, домашних заданий, материалов, комментариев и статуса оплаты.

Главная цель MVP: дать репетитору безопасный кабинет с учениками, а ученику - простой доступ по постоянной ссылке без пароля, где он видит уроки и сдает домашнее задание с фото.

Интерфейс MVP: только русский язык.

Монетизация платформы в MVP не реализуется. Финансовая часть внутри продукта ограничена отметкой "урок оплачен / не оплачен".

## 2. Роли и модель доступа

### 2.1. Репетитор

Репетитор регистрируется и входит по email и паролю.

Поля регистрации:

- имя;
- email;
- телефон;
- пароль.

Требования:

- подтверждение email при регистрации включено;
- восстановление пароля через email;
- вход по email и паролю;
- страница настроек профиля нужна.

Права репетитора:

- создавать учеников;
- редактировать карточки учеников;
- архивировать учеников;
- генерировать и перегенерировать постоянные ссылки учеников;
- копировать ссылку ученика;
- создавать, редактировать и удалять уроки;
- редактировать верхние виджеты профиля ученика;
- проверять Д/З;
- менять статус Д/З;
- оставлять комментарии к Д/З;
- отмечать оплату урока.

### 2.2. Ученик

Ученик не имеет аккаунта и пароля. Он открывает свой профиль по постоянной защищенной ссылке-токену, например:

```text
https://tutor-space.ru/student/abc123xyz
```

Требования:

- ссылка постоянная;
- репетитор может перегенерировать ссылку;
- доступ без пароля и PIN-кода;
- ссылка может открываться с нескольких устройств;
- время последнего входа ученика в MVP не показывается.

Права ученика:

- читать информацию профиля;
- открывать материалы и полезные ссылки;
- видеть уроки, Д/З, комментарии, статус проверки и оплату;
- сдавать Д/З;
- прикреплять несколько фото к Д/З;
- добавлять текстовый комментарий к сдаче;
- повторно сдавать Д/З после исправлений.

Ученик не может редактировать данные, созданные репетитором.

## 3. Персональные данные и юридические требования

Сервис обрабатывает персональные данные репетиторов, учеников и родителей.

К персональным данным относятся:

- имя репетитора;
- email репетитора;
- телефон репетитора;
- имя ученика;
- класс ученика;
- контакт ученика;
- контакт родителя;
- фото домашних заданий;
- комментарии, если в них есть сведения о человеке;
- технические данные: IP-адрес, cookies, идентификаторы аккаунтов и записей.

Решения для MVP:

- собирать только необходимые данные;
- контакт родителя сделать необязательным;
- контакт ученика сделать необязательным;
- комментарий по ученику сделать необязательным;
- стоимость и длительность занятия сделать необязательными;
- фото Д/З хранить в приватном хранилище;
- перед публичным запуском подготовить политику конфиденциальности, согласие на обработку персональных данных и пользовательское соглашение;
- финальные юридические формулировки должен проверить профильный юрист.

## 4. Карточка ученика

### 4.1. Поля карточки

Обязательные поля:

- имя ученика;
- предмет;
- статус: активный / архивный.

Необязательные поля:

- класс;
- контакт ученика;
- контакт родителя;
- стоимость занятия;
- длительность занятия;
- комментарий;
- расписание занятий;
- цели занятий;
- ссылка подключения к занятию;
- полезные ссылки.

### 4.2. Архивация

Физическое удаление ученика из базы в MVP не используется. Вместо удаления ученик переводится в архив.

История уроков, Д/З, материалов и оплат сохраняется.

## 5. Кабинет репетитора

Главная страница репетитора содержит карточки учеников.

В карточке ученика отображаются:

- имя;
- предмет;
- класс;
- расписание;
- статус активный / архивный;
- количество Д/З на проверке;
- кнопка копирования ссылки ученика.

Функции кабинета:

- поиск по ученикам;
- фильтр по статусу: активные / архивные;
- фильтр по предмету;
- фильтр по классу;
- переход в профиль ученика;
- создание нового ученика;
- копирование ссылки ученика;
- отображение бейджа "Д/З на проверке".

## 6. Профиль ученика

Профиль ученика доступен:

- репетитору из кабинета;
- ученику по постоянной ссылке.

Профиль содержит:

- верхние информационные виджеты;
- блок "Ближайшее Д/З / последнее Д/З";
- таблицу занятий;
- внутренние индикаторы обновлений.

## 7. Верхние виджеты профиля

Виджеты задаются отдельно для каждого ученика.

Список виджетов:

1. Расписание занятий.
2. Цели занятий.
3. Подключение к занятию.
4. Полезные ссылки.

### 7.1. Расписание занятий

Краткий текст, например:

```text
Пн - 16:00, Чт - 18:00
```

### 7.2. Цели занятий

Текстовое описание учебной цели, например:

```text
Подготовка к ОГЭ по математике, разбор геометрии.
```

### 7.3. Подключение к занятию

Одна внешняя ссылка, заданная репетитором.

Примеры:

- Zoom;
- Telegram;
- Skype;
- Google Meet.

### 7.4. Полезные ссылки

Список ссылок с названиями.

Примеры:

- Онлайн-доска;
- Учебник;
- Чат;
- Тренажер.

### 7.5. Поведение пустых виджетов

Для ученика пустые виджеты скрываются.

Для репетитора пустые виджеты показываются как незаполненные, чтобы репетитор мог их заполнить.

Все внешние ссылки открываются в новой вкладке.

## 8. Блок "Ближайшее Д/З / последнее Д/З"

Над таблицей занятий показывается короткий блок с актуальным домашним заданием.

Назначение:

- ученик быстро видит, что нужно сделать;
- не нужно искать актуальное Д/З в длинной истории уроков;
- таблица занятий при этом сохраняется как основная рабочая область.

Логика выбора:

- сначала показывать просроченное непроверенное Д/З;
- если просроченных несколько, показывать самое старое по дедлайну;
- если просроченных нет, показывать ближайшее непроверенное Д/З по дедлайну;
- если дедлайнов нет, показывать последнее актуальное непроверенное Д/З;
- если все Д/З проверены, можно показывать последнее проверенное или пустое состояние.

Пустое состояние должно быть спокойным и объяснять, что актуального Д/З пока нет.

## 9. Таблица занятий

Основная рабочая область профиля ученика.

Сортировка:

- новые уроки сверху.

На мобильном для ученика таблица остается горизонтально прокручиваемой.

На мобильном для репетитора используются компактные карточки уроков с действием редактирования.

### 9.1. Колонки / данные урока

Урок содержит:

- дата урока;
- тема урока;
- материалы урока;
- усвоение темы;
- домашнее задание;
- дедлайн Д/З;
- статус Д/З;
- сдачи Д/З;
- комментарии к Д/З;
- оплата;
- действия репетитора.

### 9.2. Дата

Формат отображения:

```text
ДД.ММ.ГГГГ
```

### 9.3. Тема урока

Текстовое поле с темой пройденного материала.

### 9.4. Материалы урока

Материалы урока - произвольный список внешних ссылок. Файлы репетитор в MVP не загружает.

У каждой ссылки есть:

- название;
- URL;
- тип.

Типы материалов:

- конспект;
- видео;
- презентация;
- доска;
- задачник;
- другое.

В интерфейсе длинные URL не показываются. Вместо них отображаются компактные иконки или кнопки с названием/типом материала.

Все материалы открываются в новой вкладке.

Дата и время добавления материала ученику не показываются.

### 9.5. Усвоение темы

Виджет из 5 звезд.

Значение:

- от 1 до 5;
- может быть пустым, если репетитор не указал оценку усвоения.

Управляется репетитором.

### 9.6. Домашнее задание

Текстовое описание Д/З, заданное репетитором.

### 9.7. Дедлайн Д/З

Дедлайн задается репетитором для каждого урока отдельно.

Если дедлайн не указан, просрочка не считается.

### 9.8. Статусы Д/З

Поддерживаемые статусы:

- Не сдано;
- На проверке;
- Проверено;
- Нужно исправить.

Базовая логика:

- после создания урока статус Д/З по умолчанию "Не сдано";
- после отправки Д/З учеником статус становится "На проверке";
- репетитор может поставить "Проверено";
- репетитор может поставить "Нужно исправить";
- после повторной отправки исправлений статус снова становится "На проверке".

### 9.9. Просрочка Д/З

Просрочка считается от даты дедлайна до первой отправки Д/З.

Если ученик сдал Д/З после дедлайна, система фиксирует количество просроченных дней.

Если ученик потом отправляет исправления, количество просроченных дней по первой сдаче не пересчитывается.

Если дедлайн не указан, просрочка не считается.

### 9.10. Сдача Д/З учеником

Ученик может:

- нажать "Сдать Д/З";
- прикрепить несколько фото;
- добавить текстовый комментарий;
- отправить работу;
- отправить работу повторно после исправлений.

Репетитор видит сдачу и статус "На проверке".

Для уроков со статусом "На проверке" у репетитора должна быть отдельная кнопка "Проверить Д/З".

Проверка Д/З открывается в отдельном модальном окне и содержит:

- историю сдач;
- фото/файлы ученика;
- комментарии ученика;
- поле комментария репетитора;
- выбор итогового статуса: "Проверено" или "Нужно исправить".

### 9.11. Комментарии к Д/З

Репетитор оставляет текстовый комментарий:

- разбор ошибок;
- обратную связь;
- пояснение, что нужно исправить.

Ученик видит комментарий, но не редактирует его.

### 9.12. Оплата

Оплата - простой чекбокс:

- пустой - урок не оплачен;
- отмеченный - урок оплачен.

Управляется только репетитором.

Общий долг и сумма неоплаченных занятий в MVP не показываются.

## 10. Создание и редактирование урока

Прямое редактирование ячеек таблицы заблокировано.

### 10.1. Создание урока

Репетитор нажимает "+ Добавить занятие".

Открывается модальное окно / поп-ап с формой.

Поля формы:

- дата урока;
- тема урока;
- материалы урока;
- усвоение темы;
- домашнее задание;
- дедлайн Д/З;
- комментарий к Д/З;
- оплата.

После сохранения урок добавляется в таблицу.

### 10.2. Редактирование урока

Репетитор нажимает кнопку редактирования у урока.

Открывается та же форма, заполненная текущими данными.

Репетитор может:

- изменить дату;
- изменить тему;
- изменить материалы;
- изменить рейтинг усвоения;
- изменить Д/З;
- изменить дедлайн;
- оставить или изменить комментарий;
- изменить статус Д/З;
- отметить оплату;
- удалить урок.

Удаление урока в MVP мягкое: перед удалением показывается подтверждение, после подтверждения урок скрывается из интерфейса, а история сохраняется. В будущей БД это соответствует заполнению поля `deleted_at`, без физического удаления строки.

## 11. Фото Д/З

Фото Д/З загружаются прямо в сервис.

Хранилище:

- Supabase Storage;
- приватный доступ;
- в базе хранятся пути/идентификаторы файлов.

Ограничения:

- до 10 фото на одно Д/З;
- поддерживаемые форматы: JPG, PNG, HEIC/HEIF, WebP;
- максимальный размер исходного фото до 10 МБ;
- автоматическое сжатие на клиенте перед загрузкой;
- длинная сторона после сжатия примерно 1600-2000 px;
- целевой размер после сжатия желательно 1-2 МБ на фото.

Поддерживаемые форматы:

- JPG;
- PNG;
- HEIC/HEIF при возможности браузера;
- WebP.

Если формат не поддерживается, пользователь видит понятное сообщение:

```text
Не удалось загрузить этот формат. Попробуйте выбрать JPG/PNG или сделать скриншот.
```

Ученик не должен вручную сжимать фото.

Если загрузка не удалась, система показывает понятную ошибку и позволяет попробовать еще раз.

## 12. Уведомления и индикаторы

В MVP нет email, Telegram и push-уведомлений.

Используются только внутренние индикаторы.

### 12.1. Для ученика

У ученика есть индикатор обновлений в интерфейсе.

По клику на колокольчик ученик видит короткий список 3-5 последних обновлений. После открытия списка обновления считаются просмотренными, индикатор исчезает.

События, которые считаются обновлениями:

- добавлен урок;
- изменено Д/З;
- оставлен комментарий;
- изменен статус проверки;
- добавлена ссылка на материал.

Полноценный центр уведомлений и внешние уведомления в MVP не реализуются.

### 12.2. Для репетитора

На карточке ученика показывается бейдж "Д/З на проверке", если ученик сдал работу.

Это основной MVP-индикатор для репетитора.

## 13. UI/UX и визуальный стиль

Интерфейс минималистичный, адаптированный под мобильные устройства.

Цветовая палитра:

- базовый фон страницы: `#FAFAFB`;
- структурные блоки и карточки: `#F4E8EC`;
- текст, кнопки и акценты: `#3F3F46`.

Принцип:

- 60% - базовый фон;
- 30% - структурные блоки;
- 10% - акцентные элементы.

Общие требования:

- без тяжелой Excel-сетки;
- аккуратные скругления 8-12 px;
- длинные ссылки заменяются компактными кнопками/иконками;
- основные действия должны быть заметными;
- интерфейс должен быть удобен на смартфоне.

Поля:

- имя;
- телефон;
- email;

Точный состав настроек можно уточнить на этапе проектирования.

### 14.11. Расписание

В MVP раздел "Расписание" является визуальным календарем для репетитора.

Он не создает уроки, не задает Д/З и не влияет на таблицу занятий в профиле ученика.

Полноценная связь расписания с уроками, группами, переносами и абонементами требует отдельного проектирования доменной модели и относится к Post-MVP или расширению MVP после согласования.

## 15. User stories MVP

### 15.1. Репетитор регистрируется

Как репетитор, я хочу зарегистрироваться по имени, email, телефону и паролю, чтобы получить доступ к своему кабинету.

Критерии приемки:

- после регистрации отправляется письмо подтверждения email;
- без подтверждения email пользователь видит понятное состояние;
- после входа репетитор попадает в кабинет.

### 15.2. Репетитор создает ученика

Как репетитор, я хочу создать карточку ученика, чтобы вести его уроки и домашние задания.

Критерии приемки:

- обязательные поля валидируются;
- ученик создается только внутри аккаунта текущего репетитора;
- для ученика создается постоянная ссылка доступа.

### 15.3. Репетитор копирует ссылку ученика

Как репетитор, я хочу скопировать ссылку ученика из карточки, чтобы отправить ее ученику или родителю.

Критерии приемки:

- кнопка копирует ссылку в буфер обмена;
- пользователь видит подтверждение копирования.

### 15.4. Репетитор добавляет урок

Как репетитор, я хочу добавить урок через форму, чтобы не редактировать таблицу напрямую.

Критерии приемки:

- урок появляется в таблице;
- новые уроки отображаются сверху;
- материалы сохраняются как список внешних ссылок;
- статус Д/З по умолчанию "Не сдано".

### 15.5. Ученик открывает профиль по ссылке

Как ученик, я хочу открыть профиль по постоянной ссылке без пароля, чтобы быстро увидеть уроки и Д/З.

Критерии приемки:

- ссылка открывает только профиль конкретного ученика;
- ученик не видит данные других учеников;
- ученик не может редактировать данные репетитора.

### 15.6. Ученик сдает Д/З с фото

Как ученик, я хочу прикрепить несколько фото и комментарий, чтобы отправить выполненное Д/З репетитору.

Критерии приемки:

- можно загрузить до 10 фото;
- фото автоматически сжимаются;
- после отправки статус становится "На проверке";
- репетитор видит бейдж "Д/З на проверке".

### 15.7. Репетитор проверяет Д/З

Как репетитор, я хочу увидеть отправленные фото, оставить комментарий и поменять статус, чтобы дать обратную связь ученику.

Критерии приемки:

- репетитор видит фото и комментарий ученика;
- репетитор может поставить "Проверено" или "Нужно исправить";
- ученик видит статус и комментарий.

### 15.8. Ученик отправляет исправления

Как ученик, я хочу повторно отправить Д/З после исправлений, чтобы репетитор снова проверил работу.

Критерии приемки:

- повторная отправка разрешена;
- статус снова становится "На проверке";
- история сдач не должна теряться.

### 15.9. Репетитор архивирует ученика

Как репетитор, я хочу архивировать ученика, чтобы скрыть его из активного списка, но сохранить историю.

Критерии приемки:

- ученик получает статус "архивный";
- данные не удаляются физически;
- ученик доступен через фильтр архивных.

## 16. Рекомендуемый технический стек

Фронтенд:

- React;
- адаптивная верстка;
- клиентское сжатие изображений перед загрузкой.

Бэкенд и БД:

## 18. Безопасность

Требования:

- данные каждого репетитора изолированы от других репетиторов;
- ученик по токену видит только свой профиль;
- фото Д/З хранятся приватно;
- прямые публичные ссылки на фото не должны открывать чужие файлы;
- доступ к данным должен контролироваться RLS-политиками Supabase;
- токены учеников нельзя хранить в базе в открытом виде, предпочтительно хранить хэш;
- перегенерация токена должна инвалидировать старую ссылку;
- все внешние страницы открываются через HTTPS, если URL поддерживает HTTPS.

## 19. MVP: обязательный объем

В MVP обязательно должны работать сценарии:

1. Репетитор зарегистрировался и вошел.
2. Репетитор создал ученика.
3. Репетитор получил и скопировал постоянную ссылку ученика.
4. Репетитор добавил урок.
5. Ученик открыл профиль по ссылке.
6. Ученик увидел актуальное Д/З.
7. Ученик сдал Д/З с несколькими фото и комментарием.
8. Репетитор увидел бейдж "Д/З на проверке".
9. Репетитор проверил работу, оставил комментарий и изменил статус.
10. Ученик увидел статус и комментарий.

## 20. Отложено после MVP

В MVP не входят:

- платная подписка репетитора;
- админка платформы;
- экспорт в Excel/CSV/PDF;
- email, Telegram и push-уведомления;
- полноценный центр уведомлений;
- мультиязычность;
- общий расчет долга;
- дата/время последнего входа ученика;
- загрузка файлов репетитором в материалы урока;
- лендинг;
- SMS-восстановление пароля;
- привязка одного ученика к нескольким репетиторам;
- группы учеников и групповые занятия;
- абонементы и баланс занятий;
- превращение расписания в источник создания уроков и Д/З.

## 21. Демо-данные

Для демонстрации заказчику нужны демо-данные:

- один демо-репетитор;
- 2-3 ученика;
- несколько уроков;
- материалы разных типов;
- одно Д/З на проверке;
- одно проверенное Д/З;
- одно Д/З с просрочкой;
- несколько фото-сдач, можно использовать тестовые изображения.

## 22. Принятые решения и открытые вопросы для следующего этапа

Принятые решения:

1. Расписание в MVP - только визуальный календарь. Оно не создает уроки, не задает Д/З и не влияет на таблицу занятий.
2. Удаление урока в MVP мягкое через `deleted_at`: урок скрывается из интерфейса, история сохраняется.
3. Блок "Ближайшее Д/З / последнее Д/З" выбирает задание по приоритету: просроченное непроверенное, ближайшее непроверенное с дедлайном, последнее актуальное непроверенное без дедлайна, последнее проверенное или пустое состояние.
4. Для статуса "На проверке" нужна отдельная кнопка "Проверить Д/З" и отдельное модальное окно проверки.
5. Ученик видит короткий список 3-5 последних обновлений; после открытия списка обновления считаются просмотренными.
6. Группы и абонементы остаются Post-MVP.
7. Лимиты фото Д/З: до 10 фото на сдачу, JPG/PNG/HEIC/HEIF/WebP, до 10 МБ на исходное фото, сжатие до 1600-2000 px по длинной стороне и желательно 1-2 МБ после сжатия.

Открытые вопросы:

1. Нужна ли настройка часового пояса репетитора?
2. Нужно ли ограничивать количество учеников на одного репетитора в MVP?
3. Нужно ли в MVP показывать ученику архивные уроки без ограничений или вводить пагинацию/подгрузку?
